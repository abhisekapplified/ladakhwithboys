# CLAUDE.md

Guidance for Claude Code (and other AI assistants) working in this repository.

## What this project is

A **single self-contained, offline-capable HTML page**: an interactive **9-day self-drive Ladakh
road-trip itinerary** (Delhi → Manali → Jispa → Leh → Nubra → Pangong → Leh → Sarchu → Keylong →
Manali → Shimla → Delhi). It is a static deliverable — **no build step, no framework, no backend,
no npm**. Everything (HTML, CSS, JS, SVG graphics) lives inline in one file.

- **Entry point / only real source file:** `index.html`
- **Deployed on Vercel**, served from the repo root. The file **must stay named `index.html`** so
  the bare URL (`/`) resolves — renaming it back to a descriptive name reintroduces a 404.
- **Repo:** https://github.com/abhisekapplified/ladakhwithboys (branch `main`).

## Trip assumptions baked into the content

- **Dates:** 26 Sep – 4 Oct (9 days). **Travelers:** 5. **Vehicle:** 1 self-drive SUV, ~14 km/l.
- **Budget:** mid-range; hotels costed at ~2 rooms/night; cost-per-person = total ÷ 5.
- The page is honest that the full Nubra + Pangong + **return-via-Shimla** loop is aggressive for
  9 days — it flags the rushed legs (Day 1, Day 3, Day 9) and recommends a +1-day Shimla→Delhi
  finish. **Hanle & Tso Moriri are deliberately excluded** (separate southern arm, +2 days each).
  Keep these caveats if you edit the itinerary — they are intentional expert guidance, not TODOs.

## Architecture of `index.html`

- **Data-driven.** A `const days = [ {...} ]` array near the bottom `<script>` is the **single source
  of truth** for every day. Cards, the summary table, per-day cost rows, Google-Maps links, and the
  header/backdrop photos are all generated by looping over it. **Add or edit a day by changing this
  array**, not by hand-writing markup. `passes`, `vehicleChecklist`, `packingChecklist`, and
  `dayStops` (Google-Maps waypoints) are sibling data structures.
- **Cost model.** Per-day `cost:{fuelL,hotel,food,toll,misc}` feeds the printed day totals; the live
  **cost calculator** recomputes fuel/total/per-person from its own inputs and also updates the hero
  and final-summary stat chips. If you change cost data, the calculator defaults should stay
  consistent with it.
- **Sections (in order):** hero (photo backdrop) → sticky TOC nav → illustrated SVG circuit map →
  feasibility "reality check" → day-wise summary table → collapsible day cards → mountain-pass
  reference → cost calculator → safety → vehicle checklist → packing checklist → final summary →
  footer disclaimer.
- **Each day card** is a `<details>` whose `<summary>` is a full-width **photo banner** (the day's
  lead image as a cover background with a gradient overlay) and whose body uses the same photo as a
  faint backdrop behind translucent field cards. There is **no thumbnail gallery** — that was removed
  intentionally in favour of the backdrop treatment. A lightbox helper still exists but is currently
  inert (no callers); leave or remove it deliberately.
- **The map is pure inline SVG/CSS/SMIL** (no image tiles, no Leaflet): layered mountains with snow
  caps + haze, gradient glowing routes with animated flow + arrows, glowing pin nodes, a sunrise/
  aurora/sun sky, twinkling stars, pass glyphs, a compass, and a **red SUV that drives the full loop**
  via `<animateMotion><mpath href="#loop"/>`. The hidden `#loop` path must trace the same node order
  as the visible road segments, or the SUV drifts off the roads.

## Images

- All photos are **verified Wikimedia Commons thumbnail URLs** (each returns HTTP 200). When adding
  images, fetch real URLs from the Commons API and confirm `200` before wiring them in — do **not**
  hand-guess `upload.wikimedia.org/.../<hash>/...` paths (the hash is unpredictable and 400s).
  Working pattern:
  ```
  https://commons.wikimedia.org/w/api.php?action=query&generator=search&gsrsearch=<place>&gsrnamespace=6&gsrlimit=3&prop=imageinfo&iiprop=url&iiurlwidth=960&format=json
  ```
  (send a descriptive `User-Agent` and space requests out — the API rate-limits aggressive loops).
- Each `PIC(url, caption)` entry's first photo is the day's banner/backdrop. An `onerror` placeholder
  keeps the page intact if an image ever fails offline.

## Conventions

- **Keep it one file, offline-first.** No external JS/CSS, no CDN, no fonts beyond system stack.
  Remote images may fail without a signal; text, tables, cards, calculator and the SVG map must all
  still work. Don't introduce a build tool.
- **Responsive.** Breakpoints at 768 / 560 / 380 px. Test layout narrow (mobile) and via the
  browser's Print preview (there is a print stylesheet) after layout changes.
- Match the existing dark "Himalaya" theme — CSS custom properties are defined in `:root`.

## Local preview & verification

There is no dev server. To preview/screenshot, serve the folder statically, e.g.:

```bash
python -m http.server 8099 -d .
# then open http://localhost:8099/
```

After any change, sanity-check in a browser: page renders with **no console errors**; all 9 day
cards expand/collapse; the cost calculator updates total & per-person live; Google-Maps buttons carry
valid `https://www.google.com/maps/dir/?...` URLs; the SVG map renders and the SUV animates.
Note: continuous SVG animations can make headless screenshot tools time out — that's a capture quirk,
not a page bug; pause animations or reset the preview if needed.

## Git / deploy

- Branch `main`, remote `origin` → GitHub repo above. **Commit only when asked**; push triggers a
  Vercel redeploy automatically. Never force-push `main`.
- Imperative commit subject + short body explaining the "why". Co-authorship trailer is fine.
- Don't commit local tooling (`.claude/`, editor files). The repo also contains a default `README.md`.
